# TA Recruitment System User Manual

## 1. Purpose

This manual explains how to run, test, and use the TA Recruitment System. It is written for project demonstrations, coursework assessment, testers, and first-time users.

System name: TA Recruitment System  
Default context path: `/SE`  
Default TA entry URL: `http://localhost:8081/SE/ta`

## 2. System Overview

The TA Recruitment System is a JSP/Servlet web application for managing teaching assistant recruitment at BUPT International School.

The system supports three user roles:

- **TA**: browses jobs, registers with a BUPT email, uploads a resume, generates and submits an application form, manages applications, withdraws applications, and checks review results.
- **MO**: manages assigned course projects, edits and publishes recruitment information, reviews applications, saves selected candidates, and publishes final review results.
- **Admin**: creates MO accounts, assigns courses, views candidates, sets deadlines, and resets the yearly recruitment cycle.

The project uses text-file persistence instead of a database. This keeps deployment simple and makes the system easy to demonstrate for coursework.

## 5. Architecture

The project follows this structure:

```text
JSP page
  -> Servlet Controller
  -> Service
  -> Repository
  -> Store
  -> text file
```

Layer responsibilities:

| Layer | Responsibility |
|---|---|
| JSP | Renders pages and forms |
| Controller | Handles requests, checks sessions and roles, forwards or redirects |
| Service | Implements business rules such as deadlines, application limits, and review publication |
| Repository | Provides data access interfaces |
| Store | Reads and writes local text files |
| Model | Represents users, courses, resumes, and application forms |

The recent CSV persistence fix was implemented inside the Store layer only. Repository interfaces and repository-to-store calls were not changed.

Deadline design:

- Admin saves the TA application deadline into `applicationDeadline` in `ServletContext`.
- Admin saves the MO course modification deadline into `moCourseModifyDeadline` in `ServletContext`.
- TA and MO pages read deadline status from `ServletContext`.
- `AppContextListener` rebuilds these context values when the web application starts by loading `deadline.txt` and `mo-deadline.txt` from the deployed `WEB-INF/file` directory.
- Store classes still handle persistence. `ServletContext` is the runtime source used by page controllers.

Review notification design:

- When an MO publishes final review results, affected TA applications are marked as unread.
- A logged-in TA can see a red notification dot on the **Personal Center** link on the TA public page.
- In **Personal Centre**, each course with a newly published result also shows a red dot.
- Opening **Personal Centre** clears all unread review notifications for that TA.
- Guests and logged-out users do not see notification dots.

Annual reset design:

- Admin can use **Reset Recruitment Cycle** at the beginning of a new recruitment year.
- The reset clears TA applications, generated application forms, MO review picks, final review results, unread review notifications, and both deadline values.
- It keeps Admin/MO/TA accounts, MO-course ownership, course names, and the previous recruitment text.
- Every course returns to Draft, so MOs can reuse and edit the previous text before publishing again.

| `courses.txt` | Stores courses and recruitment information |

| `mo-deadline.txt` | Stores the MO course modification deadline |
| `application-forms.txt` | Stores generated and submitted TA application forms |
| `resume/` | Stores uploaded PDF resumes |

### 6.1 Reset Runtime Data for Manual Testing

If you want to clear the running system and test everything manually from the beginning, clear the Tomcat runtime data folder:

```text
TOMCAT_HOME/webapps/SE/WEB-INF/file/
```

On the default Windows setup, this is:

```text
G:\Tomcat\webapps\SE\WEB-INF\file\
```

Clear these files or empty their contents:

| File or Folder | What It Resets |
|---|---|
| `users.txt` | User accounts, TA registration data, application status linked to users |
| `courses.txt` | Courses, MO assignments, job information, draft/published status |
| `application-forms.txt` | Generated and submitted TA application forms |
| `deadline.txt` | TA application deadline |
| `mo-deadline.txt` | MO course modification deadline |
| `resume/` | Uploaded PDF resumes |

Optional files:

| File | Note |
|---|---|
| `candidates.txt` | Legacy or empty candidate data, safe to clear for a clean demo |
| `groq-transport.log` | AI transport log, safe to clear |

Recommendation:

- Keep the file names and folder names.
- Empty file contents instead of deleting the files.
- Empty the contents of `resume/` but keep the folder.
- Restart Tomcat or run `start_SE.bat` again after clearing deadline files, because deadline status is loaded into `ServletContext` when the web application starts.

The source folder also contains template data:

```text
webapp/WEB-INF/file/
```

Usually, you only need to clear the Tomcat runtime folder. Clear the source folder only if you want future redeployments to start from a clean template too.

### 6.2 CSV Persistence Fix

The project previously split `users.txt` and `courses.txt` rows directly by commas. That could corrupt data when a value contained a comma, such as:

```text
Java, Python, SQL
```

The Store layer now uses standard CSV escaping:

```text
"Java, Python, SQL"
```

Values containing double quotes are escaped by doubling the inner quote:

```text
"TA ""Lead"""
```

The value is restored correctly when loaded.

Files involved in this fix:

- `webapp/WEB-INF/src/store/CsvRecord.java`
- `webapp/WEB-INF/src/store/CourseStore.java`
- `webapp/WEB-INF/src/store/UserStore.java`
- `src/test/java/store/CourseStoreTest.java`
- `src/test/java/store/UserStoreTest.java`

## 9. Optional AI Key Setup

The system can use Groq API to generate TA application forms and initial MO course recruitment drafts.

If no API key is configured, the system still works. It falls back to mock/default generated content. However, for the best demo experience, set a Groq API key on your own computer before starting Tomcat.

### 9.1 Required Environment Variables

Set these variables:

```text
GROQ_API_KEY=your_groq_api_key_here
GROQ_MODEL=llama-3.1-8b-instant
```

`GROQ_MODEL` is optional. If it is not set, the project uses:

```text
llama-3.1-8b-instant
```

### 9.2 Windows Setup

Recommended method: set user-level environment variables in PowerShell.

Open PowerShell and run:

```powershell
[Environment]::SetEnvironmentVariable("GROQ_API_KEY", "your_groq_api_key_here", "User")
[Environment]::SetEnvironmentVariable("GROQ_MODEL", "llama-3.1-8b-instant", "User")
```

Then close and reopen PowerShell, Command Prompt, IntelliJ, VS Code, or any terminal that will start Tomcat.

To check whether the key is available:

```powershell
[Environment]::GetEnvironmentVariable("GROQ_API_KEY", "User")
[Environment]::GetEnvironmentVariable("GROQ_MODEL", "User")
```

The Windows deployment script `start_SE.bat` attempts to load `GROQ_API_KEY` and `GROQ_MODEL` from the user environment before starting Tomcat.

Temporary PowerShell-only setup:

```powershell
$env:GROQ_API_KEY="your_groq_api_key_here"
$env:GROQ_MODEL="llama-3.1-8b-instant"
.\start_SE.bat
```

This temporary method works only for the current PowerShell session.

### 9.3 macOS/Linux Setup

Temporary terminal setup:

```bash
export GROQ_API_KEY="your_groq_api_key_here"
export GROQ_MODEL="llama-3.1-8b-instant"
./start_se.sh
```

Persistent setup for zsh:

```bash
echo 'export GROQ_API_KEY="your_groq_api_key_here"' >> ~/.zshrc
echo 'export GROQ_MODEL="llama-3.1-8b-instant"' >> ~/.zshrc
source ~/.zshrc
```

Persistent setup for bash:

```bash
echo 'export GROQ_API_KEY="your_groq_api_key_here"' >> ~/.bashrc
echo 'export GROQ_MODEL="llama-3.1-8b-instant"' >> ~/.bashrc
source ~/.bashrc
```

### 9.4 How to Verify AI Setup

After starting the system:

1. Log in as a TA.
2. Upload a PDF resume.
3. Open a published course.
4. Click **Generate Standard Form**.
5. Check whether the generated form is tailored to the resume and course.

For MO course draft generation:

1. Log in as Admin.
2. Open **MO Accounts**.
3. Create a new MO and enter a new course name.
4. Check whether the generated draft contains course-specific job description and requirements.

If the API key is missing or invalid, the system should still continue with fallback content.

### 9.5 Security Notes

- Do not commit your real API key to Git.
- Do not paste the API key into `users.txt`, `courses.txt`, JSP pages, or Java source files.
- Use environment variables instead.
- If a key is accidentally exposed, revoke it from the Groq dashboard and create a new one.

## 11. TA Workflow

### 11.1 Enter the TA Page

Open:

```text
http://localhost:8081/SE/ta
```

The TA home page provides job browsing, personal centre access, login/register links, and a recruitment timeline.

The recruitment timeline shows:

- TA Application Deadline
- Course Information Deadline
- Whether TA application is open or closed
- Whether MO recruitment editing is open or locked
- Whether the review stage is open

This information is visible even before a TA logs in, so a visitor can understand the current recruitment status before registering or applying.

### 11.2 Register as a TA

1. Click **TA Login/Register**.
2. Enter a BUPT email address, for example:

```text
student001@bupt.edu.cn
```

3. Submit the registration form.
4. The system displays a generated access key, such as:

```text
TA-ABCD-2345
```

5. Save the access key.

Rules:

- The email must end with `@bupt.edu.cn`.
- A registered email cannot be used again.
- The generated access key is the TA login credential.

### 11.3 Log in as a TA

1. Open the TA Login/Register page.
2. Enter the access key.
3. After successful login, the system returns to the TA job page.

### 11.4 Browse Jobs

1. Click **Find a Job**.
2. The system shows published TA positions.
3. Click a course card to open the course detail page.

The course detail page shows:

- Course name
- Job title
- Job requirement
- Generate or modify application form
- Withdraw application, if the TA has already applied

### 11.5 Upload a Resume

Before applying, a TA must upload a PDF resume.

1. Open a course application page.
2. Choose a PDF file.
3. Upload it.
4. The uploaded file becomes the TA profile resume.

Rules:

- Only PDF files are accepted.
- If a TA already has a profile resume, the same resume is reused for applications.
- To replace the resume, use the Personal Centre.

### 11.6 Generate an Application Form

1. Open a course detail page.
2. Click **Generate Standard Form**.
3. The system uses the TA profile, resume text, and course information.
4. If Groq API is configured, the system attempts to generate a more detailed AI-assisted form.
5. If AI is unavailable, fallback generation is used.

Generated fields include:

- Applicant name
- Email
- Education
- Skills
- Relevant experience
- Project experience
- Feedback

### 11.7 Submit an Application

1. Review the generated form.
2. Edit any fields if necessary.
3. Click **Submit Application**.
4. The system returns to the course detail page and shows a success message.

Rules:

- A TA can apply for up to 3 different positions.
- After the TA application deadline, new applications, modifications, and withdrawals are disabled.

### 11.8 Modify an Application

Before the deadline:

1. Open **Personal Centre**.
2. Select an applied course.
3. Click **Modify Application Form**.
4. Edit and submit again.

### 11.9 Withdraw an Application

Before the deadline:

1. Open **Personal Centre** or the course detail page.
2. Find the applied course.
3. Click **Withdraw**.
4. Confirm the withdrawal.

After withdrawal, the course is removed from the TA application list.

### 11.10 Check Review Results

After an MO publishes review results:

1. Open **Personal Centre**.
2. Select the applied course.
3. Check the status:
   - Pending
   - Approved
   - Rejected

If there is a newly published result, the logged-in TA public page shows a red notification dot on **Personal Center**. In **Personal Centre**, the updated course also shows a red dot. Once Personal Centre is opened, all unread result notifications for that TA are cleared.

### 12.2 MO Dashboard

- Personal Centre

The dashboard also shows a recruitment timeline:

- TA Application Deadline
- Course Information Deadline
- TA application status
- MO recruitment editing status
- Review stage status

If the MO course modification deadline has passed, Publish Recruitment is locked and a deadline message is shown.

### 12.3 View My Project

1. Click **Publish Recruitment** or **My Project**.
2. The system lists courses assigned to the current MO.
3. Click a course to open Project Detail.
4. Use **Back to Personal Centre** to return to the MO personal centre.
5. Use **Back to Dashboard** to return to the MO dashboard.

### 12.4 Edit Recruitment Information

In Project Detail, the MO can edit:

- Job title
- Job description
- Job requirement

Actions:

- **Save draft**: saves changes without publishing the position to TAs.
- **Publish recruitment**: publishes the position so TAs can see it.

Rules:

- Editing and publishing are disabled after the MO course modification deadline.
- An MO can only manage assigned courses.

### 12.5 Review Applications

Review opens after the TA application deadline has passed.

1. Open **Personal Centre**.
2. Click **Review applications**.
3. Choose a course.
4. Review submitted application forms.
5. Use **Back to Personal Centre** to leave the review page.

The review table shows:

- Applicant name
- Email
- Education
- Skills
- Relevant experience
- Project experience
- Selection checkbox

### 12.6 Save Picks

1. Select candidates.
2. Click **Save Picks**.
3. The selection is saved but not published to TAs.

This is useful before final confirmation.

### 12.7 Publish Review Results

1. Select final accepted candidates.
2. Click **Publish Review**.
3. Selected candidates become Approved.
4. Other submitted candidates become Rejected.
5. TAs can view results in Personal Centre.
6. Affected TAs receive unread review-result notifications.
7. After publication, the course review result cannot be edited.

The MO Accounts page lists:

- MO account
- Password
- Assigned courses

### 13.5 View Candidates

1. Open **Candidates**.
2. View registered TA users.
3. Check their access keys, applied courses, and course information.
4. If a resume exists, open it from the candidate workflow.

### 13.6 Set the TA Application Deadline

1. Open **TA Application Deadline**.
2. Enter date and time.
3. Save.

Effects:

- Before the deadline, TAs can apply, modify, and withdraw.
- After the deadline, TA changes are locked.
- After the deadline, MO review opens.
- The value is saved to `deadline.txt` and stored in `ServletContext` as `applicationDeadline`.
- The TA public page and MO dashboard display the deadline and status.

### 13.7 Set the MO Course Modification Deadline

1. Open **MO Course Modification Deadline**.
2. Enter date and time.
3. Save.

Effects:

- Before the deadline, MOs can edit and publish recruitment information.
- After the deadline, MO editing and publishing are locked.
- The value is saved to `mo-deadline.txt` and stored in `ServletContext` as `moCourseModifyDeadline`.
- The TA public page and MO dashboard display the deadline and status.

### 13.8 Reset the Recruitment Cycle

Use this feature at the beginning of a new yearly recruitment cycle.

1. Open **Reset Recruitment Cycle** from the Admin Dashboard.
2. Read the reset summary.
3. Type:

```text
RESET
```

4. Click **Reset Cycle**.

Effects:

- TA accounts remain available.
- MO accounts remain available.
- MO-course ownership remains unchanged.
- Course names and previous recruitment text remain.
- All courses become Draft.
- TA applications are cleared.
- Generated application forms are cleared.
- MO review picks and final review results are cleared.
- Unread review notification dots are cleared.
- TA and MO deadline values are cleared.

## 16. Common Problems

### 16.1 Page Cannot Be Opened

Check:

1. Tomcat is running.
2. Port 8081 is used.
3. The URL includes `/SE`, such as `/SE/ta`.
4. `TOMCAT_HOME` points to the correct Tomcat folder.

### 16.2 Login Fails

Check:

1. Email and password are correct.
2. TA users log in with an access key, not an email/password pair.
3. `users.txt` exists.
4. The Tomcat runtime data file is the one being inspected.

### 16.3 TA Cannot See Courses

Possible reasons:

1. No MO has published recruitment yet.
2. The course is still a draft.
3. Tomcat is using a different `courses.txt` file from the one being edited.

### 16.4 MO Cannot Edit Courses

Possible reasons:

1. The MO course modification deadline has passed.
2. The account does not own the course.
3. The session has expired.
4. The deadline file was cleared while Tomcat was still running. Restart Tomcat so `ServletContext` is rebuilt.

### 16.5 MO Cannot Review Applications

Review opens only after the TA application deadline has passed.

### 16.6 TA Cannot Apply or Modify

Possible reasons:

1. The TA application deadline has passed.
2. The TA already applied for 3 positions.
3. The uploaded resume is not a PDF.
4. The session has expired.
5. The deadline file was changed while Tomcat was still running. Restart Tomcat so `ServletContext` is rebuilt.

### 16.7 AI Generation Does Not Work

Check:

1. `GROQ_API_KEY` is set on the computer that starts Tomcat.
2. Tomcat was restarted after setting the environment variable.
3. The key has not expired or been revoked.
4. Network access to Groq API is available.
5. The system can still use fallback generation if the AI call fails.

### 16.8 TA Does Not See a Review Notification Dot

Check:

1. The TA is logged in. Guests do not see notification dots.
2. The MO has published final review results, not only saved picks.
3. The TA has not already opened Personal Centre after publication.
4. Refresh `/SE/ta`. The TA entry page refreshes the logged-in TA session from runtime storage before rendering.

## 19. Known Limitations

This is a coursework prototype rather than a production system.

Known limitations:

- Passwords are still stored in text files. A real system should hash passwords.
- Some JSP pages still directly output values. A real system should apply consistent HTML escaping.
- Runtime paths assume the deployed context is `/SE`.
- Groq API transport currently depends on Windows PowerShell.
- File-based persistence has limited concurrency protection.

These limitations do not block the current coursework demo, but they should be prioritized in future development.
